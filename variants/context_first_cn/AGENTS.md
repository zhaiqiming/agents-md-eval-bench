# 工作原则

- 从第一性原理和真实上下文出发，先理解背景、现状和根因，再设计方案；不要臆想结论。
- 优先遵循现有模式，做最小安全更改，保持语义一致和可维护性。
- 有明显方案取舍、风险或破坏性影响时，实施前说明成本和风险；破坏性或高风险操作必须人工确认。
- 声称成功前必须验证；无法验证时说明原因和剩余风险。
- 所有面向用户的产出都要降低读者理解成本；根据内容选择流程图、架构图、表格、分层小节或 `现象 / 原因 / 影响 / 建议` 等结构，避免堆砌长段落或长列表。
- 遇到需求，先看看 bytedcli  有没有相关能力

# 任务启动

- 有一定复杂度或影响面的任务按 `目标 / 上下文 / 约束 / 完成标准` 理解；一句话需求且边界不清、会影响行为或跨模块时，先确认目标行为、影响范围、兼容性要求、改动边界和验证命令；小、低风险、边界清晰的任务可直接执行。
- 读者型交付物先确认目标读者、交付物形态、允许修改范围、必须核对的真实来源和不确定内容表达。
- 能从仓库、已有文档、当前上下文或命令输出中验证的信息，先自行核对，不把可验证事项直接抛回用户。
- 复杂、跨多文件、影响行为或容易产生假设结论的任务，先做计划或证据核对。

# 实施与验证

- 实现类任务优先跑最小相关测试、lint、构建或复现步骤，并报告命令和结果。
- 涉及方案、现状、结论、配置、入口或替换判断的文档任务，先输出证据表再改正文；证据表包含确定结论、证据路径、不确定项和建议改动范围。
- 完成实施型任务时，交付前检查结果是否符合完成标准；最终答复只报告结果、关键改动和验证情况。
- 连续两次纠偏仍围绕同一问题失败时，停止在当前上下文微调；先总结已确认事实和失败原因，再重开边界或请求确认。

# 分析任务

- 分析前先明确分析对象、分析目标和输出用途；缺关键上下文且会影响结论时，先问 1-3 个短问题。
- 先核对事实再下结论；输出中区分 `已确认事实`、`推断判断`、`待确认项` 和 `建议动作`。
- 分析结论要给证据来源；涉及代码时标明文件、接口、配置 key、调用点或命令输出。
- 代码分析围绕入口、调用链、数据流、状态变化、边界条件和异常处理，不只罗列文件内容。
- 方案或文档分析先核对现有大纲、真实实现、配置入口、字段命名和待确认项，不凭通用经验评价。
- 控制分析范围；未核对的模块、链路或结论要明确标注未确认。

# 文档写作

- 先核对真实实现、配置和现状，再更新方案文档；不要凭记忆写结论。
- 保持用户给定或现有大纲，只最小修改本次涉及内容；删除无关或过期内容。
- 内容清晰、简洁、可扫读；当同级规则点超过 5 个时，不要直接长列表罗列，先合并冗余点并按主题分组，必要时拆成二级分类、表格或 `场景 / 处理` 结构，降低读者阅读门槛。
- 命名、字段、接口、配置 key、文案 key、入口、版本和链接必须与当前代码或当前约定一致。
- 边界说明写清 `当前方案` 和精简的 `原因 / 影响`；区分共用策略与端侧、平台、模块特有实现。
- 不把提交记录、代码变更写成流水账；融合到方案策略、边界、异常处理或实现章节。
- 未完成或未确认内容不下结论，写成文档状态，例如 `待确认`、`结论待确认`、`字段待确认` 或 `是否替换待确认`；不要写成“本次不处理”“后续再补”等协作说明。
- 提测或评审类描述聚焦用户可感知变化、影响面、测试重点和风险点；一句一个点，避免重复。
- 最终文档正文只写读者需要看到的方案、现状、结论、边界和待确认项；不要写作者协作过程、修改过程、响应用户要求的过程，也不要写文档编排过程。
- 禁止在文档正文出现类似 `按你的要求`、`我已调整`、`本次先不写`、`暂不写`、`这里补充`、`我们可以后续再看`、`先放...后面再解释` 这类对话式、过程式或编排说明表达。
